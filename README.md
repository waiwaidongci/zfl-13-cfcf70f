# 潮汐池生态模拟

静态浏览器小游戏，直接打开 `index.html` 即可运行。

玩家通过放置岩缝、海藻、贝群和遮阴来维持潮间带生态稳定。系统包含潮汐循环、物种数量变化、捕食关系、低水位压力、生态稳定度评分和生态预算机制。

## 快速开始

### 方式一：直接打开 HTML

无需安装依赖，直接用浏览器打开 `index.html` 即可游玩：

```bash
open index.html
```

### 方式二：使用本地开发服务器

首次接手项目时先安装开发依赖：

```bash
npm install
```

启动本地静态服务器：

```bash
npm run dev
```

默认访问地址为 `http://127.0.0.1:8080`。如需只启动服务但不自动打开浏览器，可使用：

```bash
npm run serve
```

## 模块架构

为了便于后续扩展玩法，代码已拆分为清晰的分层模块。所有模块使用 UMD 封装，静态页面可直接打开运行，无需构建工具。

### 文件结构

| 文件           | 模块名              | 职责说明                                                                                         |
| -------------- | ------------------- | ------------------------------------------------------------------------------------------------ |
| `core.js`      | `TidepoolCore`      | 纯计算核心：潮位计算、稳定度评分、物种统计、变化解释、邻格查找等（无 DOM 依赖，可 Node.js 测试） |
| `state.js`     | `TidepoolState`     | 状态模型：grid 地图、tick/day/budget、挑战进度、反馈历史、序列化/反序列化、沙盘存储工具          |
| `ecosystem.js` | `TidepoolEcosystem` | 生态推进规则：工具应用（applyTool）、半日推进（advance）、挑战目标评估、开始/退出挑战            |
| `renderer.js`  | `TidepoolRenderer`  | Canvas 绘制层：绘制格子、动物、辅助图形、坐标换算、点击命中检测                                  |
| `ui.js`        | `TidepoolUI`        | DOM 更新与事件绑定：面板刷新、反馈面板、挑战面板、沙盘管理、所有事件监听器                       |
| `app.js`       | —                   | 主入口：监听 `DOMContentLoaded`，调用 `TidepoolUI.initUI()` 启动应用                             |
| `styles.css`   | —                   | 样式表（未改动，保持原有外观）                                                                   |
| `index.html`   | —                   | 页面结构（仅更新了 `<script>` 引用顺序）                                                         |
| `test.js`      | —                   | core.js 的 Node.js 单元测试                                                                      |

### 依赖关系（加载顺序）

```
core.js → state.js → ecosystem.js → renderer.js → ui.js → app.js
```

- 每个模块通过 IIFE + UMD 模式注册到全局 `window`
- 下层模块不直接引用上层，避免循环依赖
- `core.js` 保持纯计算纯净，所有带副作用的操作均在 `ecosystem.js` 及以上层

## 生态反馈面板

右侧面板中部的「生态反馈面板」将原本随机、黑盒式的生态演化过程转化为可解释的透明反馈。每次推进半日后，面板会自动分析稳定度变化的主要驱动因素，帮助玩家理解自己的操作如何影响系统。

### 面板结构

| 区域             | 说明                                                                        |
| ---------------- | --------------------------------------------------------------------------- |
| **Δ 指示器**     | 右上角显示本轮稳定度的绝对变化，▲绿色为上升，▼红色为下降，—灰色为持平       |
| **摘要语句**     | 一行可读文字概括本轮变化主因，例如「遮阴保护推动稳定度上升 3 点」           |
| **影响因素标签** | 带强度条（三格）的彩色标签，显示检测到的所有显著因素及其贡献值              |
| **最近轮次影响** | 最多保留最近 8 轮的时间线卡片，包含天数、潮期、稳定度分数、变化值和原因标签 |

### 影响因素分类（6 类）

系统通过对比推进前后的物种数据、庇护所覆盖、潮位条件以及稳定度各分项（庇护、食物、拥挤惩罚、平衡分）的差值，自动将本轮变化归类为以下 6 种可解释因素：

| 因素         | 极性         | 触发条件（示例）                                                | 建议策略                   |
| ------------ | ------------ | --------------------------------------------------------------- | -------------------------- |
| **低潮失水** | 负面（红）   | 潮位 ≤ 28% 且螺/贝因裸露失水死亡，或潮位极低且遮阴/岩缝覆盖不足 | 在高地沙格多放遮阴和岩缝   |
| **遮阴保护** | 正面（蓝）   | 新增遮阴或岩缝导致庇护分项提升，或低潮时有足够遮阴且螺类存活    | 低潮前在暴露区域补遮阴     |
| **海藻扩张** | 正面（绿）   | 海藻数量增长带动食物分项或螺类种群同步增加                      | 在湿格周围链式扩散海藻     |
| **捕食压力** | 负面（粉）   | 海星捕食贝类或小蟹捕食螺类造成损失，或海星新个体出生            | 贝群过密时先清理再放岩缝   |
| **贝类过密** | 负面（蓝灰） | 贝类超过 28 只触发拥挤惩罚，或数量继续增加导致惩罚加深          | 贝群密度控制在 18–26 区间  |
| **物种平衡** | 中性（金）   | 螺-藻、海星-贝的比例失衡改善或恶化，影响平衡分项                | 注意螺类≈3×海藻、贝≈4×海星 |

### 与潮汐事件的区别

- **生态反馈面板**聚焦于**稳定度的驱动原因**（因果关系、量化贡献），每轮必更新，避免与「潮汐事件」的叙述性描述堆叠。
- **潮汐事件列表**仍保留，用于记录叙事性现象（满潮养分、生物压力、螺类出生等）。
- 两者信息互补：面板告诉你「为什么分数变了」，事件列表告诉你「发生了什么事」。

### 评分逻辑透明化

稳定度分项（由 `stabilityComponents` 输出）构成可追踪的评分骨架：

```
总分 = 基础(28) + 庇护分项 + 食物分项 + 平衡分项 − 拥挤惩罚
```

- **庇护分项**：`min(22, 岩缝×1.2 + 遮阴×1.6)` — 遮阴权重更高
- **食物分项**：`min(26, 海藻×3 + 贝类×0.8)` — 海藻是主要食物源
- **平衡分项**：`24 − |螺−3×海藻|×0.7 − |4×海星−贝|×0.42`
- **拥挤惩罚**：`max(0, 贝−28)×1.4 + max(0, 蟹−12)×2`

每次「推进半日」后，`explainStabilityChange` 会对比两轮的分项差与种群差，结合当前潮位，自动匹配到上述 6 类因素，并按 |Δ| 排序输出，生成摘要与标签。

## 生态预算系统

每次放置或移除调控元素都会消耗或返还预算，初始预算为 40。右侧面板顶部显示当前剩余预算与进度条，预算低于 10 时进度条变红警示。

各工具的成本（放置消耗 / 移除回收）：

| 工具 | 成本 | 说明                               |
| ---- | ---- | ---------------------------------- |
| 岩缝 | 3    | 为生物提供遮蔽，提升稳定度         |
| 海藻 | 2    | 提供食物来源，吸引螺类             |
| 贝群 | 4    | 提供养分，吸引海星捕食             |
| 遮阴 | 1    | 减缓低潮时的生物压力               |
| 清理 | 回收 | 移除格子上所有元素并按原价返还预算 |

**预算不足时**放置操作会被阻止，画布下方会弹出红色提示「预算不足！」。合理规划放置顺序、及时移除不需要的元素回收预算是重要策略。

## 任务挑战模式

点击顶栏右上角的「任务挑战」按钮即可进入挑战选择界面。当前提供三个挑战：

- **稳定守护者**：在第6天结束前，将生态稳定度维持在80以上。
- **贝类平衡师**：在第5天到第8天期间，贝类数量保持在18~28之间。
- **低潮抗压**：在10天内，累计低潮压力事件不超过6次。

进入挑战后，右侧面板顶部会出现挑战进度区，实时展示各目标的完成状态（黄色待完成、绿色已达成、红色失败）。挑战完成或失败时，画布上方会弹出状态提示。

**重置按钮行为**：在挑战中点击「重置」会将潮汐池恢复到该挑战的初始状态（第1天、物种与工具配置归零、预算恢复至 40），而不会退出当前挑战。点击进度区底部的「退出挑战」可回到自由模式。

## 沙盘方案管理

点击顶栏右上角的「沙盘方案」按钮即可打开方案管理面板。玩家可以将当前潮汐池状态保存到浏览器本地存储（localStorage），随时加载、覆盖或删除已有方案。

### 保存内容

每个方案会完整保存以下游戏状态：

- **潮汐池布局**：所有格子的地形（岩缝、海藻、贝群、遮阴）和物种分布
- **物种数量**：螺类、小蟹、贝类、海星的具体数量
- **进度状态**：当前天数、潮汐 tick（潮位）
- **生态预算**：剩余预算值
- **当前工具**：工具栏选中的工具
- **事件日志**：最近 5 条潮汐事件记录
- **挑战状态**：如果正在进行挑战，会同步保存挑战进度（含压力次数、完成/失败状态）

### 操作说明

| 操作       | 说明                                                                   |
| ---------- | ---------------------------------------------------------------------- |
| 保存新方案 | 在输入框中填写名称，点击「保存当前状态」。若名称已存在则会覆盖原有方案 |
| 加载方案   | 点击方案卡片上的「加载」按钮，立即恢复到该方案的状态，模态框自动关闭   |
| 覆盖方案   | 点击「覆盖」按钮，用当前游戏状态更新该已保存方案                       |
| 删除方案   | 点击「删除」按钮，确认后从本地存储中移除该方案                         |

### 自动推进状态一致性

为保证状态一致性，系统会自动处理自动推进与其他操作的交互：

- **打开沙盘面板时**：如果自动推进正在运行，会自动暂停，确保保存的是静止状态
- **保存/覆盖时**：如果自动推进正在运行，会先暂停再保存，避免中间状态
- **加载方案后**：自动推进始终处于暂停状态（按钮文字为「自动推进」），玩家可手动决定是否继续
- **挑战状态联动**：加载方案时，挑战进度面板、成功/失败提示会同步恢复
- **重置模拟**：点击重置会立即停止自动推进并恢复初始状态
- **挑战完成/失败**：自动停止推进，避免后续状态继续变化

#### 生命周期管理

自动推进的状态通过以下统一函数管理，确保所有操作入口的一致性：

| 函数                  | 作用                                               |
| --------------------- | -------------------------------------------------- |
| `startAutoAdvance()`  | 启动自动推进，设置定时器和按钮状态，包含执行锁保护 |
| `stopAutoAdvance()`   | 停止自动推进，清除定时器，重置按钮状态与无障碍属性 |
| `toggleAutoAdvance()` | 切换推进状态，内部调用上述两个函数                 |
| `isAutoRunning()`     | 查询当前是否处于自动推进状态                       |

所有状态转换点（重置、退出挑战、挑战完成/失败、加载沙盘、打开沙盘、保存/覆盖沙盘）都会调用 `stopAutoAdvance()`，从根源上避免旧定时器继续运行。

#### 执行锁机制

`advance()` 函数通过 `isAdvancing` 标志位实现互斥执行：

- 如果当前正在执行推进逻辑，后续调用会直接返回
- 重置和加载沙盘时会强制重置锁标志位，避免死锁
- 这保证了自动推进定时器和手动点击「推进半日」不会并发导致状态错乱

### 自动推进状态一致性验证步骤

以下步骤用于验证自动推进在各种场景下的状态一致性：

#### 场景一：自动推进中快速点击重置

1. 打开 `index.html`
2. 点击「自动推进」按钮，确认按钮文字变为「暂停推进」，按钮背景变为深蓝色并有脉冲圆点动画
3. 在自动推进运行期间（至少等待一次推进，即 0.9 秒），连续快速点击「重置」按钮 3-5 次
4. **验证点**：
   - 自动推进立即停止，按钮文字恢复为「自动推进」
   - 按钮 `aria-pressed` 属性为 `"false"`
   - 天数显示为「第1天」，不再继续增加
   - 事件日志清空，只显示「潮汐池进入初始观察。」
   - 反馈面板显示「等待推进观察...」
   - 稳定度恢复到初始值

#### 场景二：自动推进中切换工具与操作格子

1. 启动自动推进
2. 快速依次执行：选择「岩缝」工具 → 点击多个沙格子 → 选择「海藻」工具 → 点击格子 → 选择「清理」工具 → 点击格子
3. **验证点**：
   - 自动推进继续运行，按钮状态保持「暂停推进」
   - 所有工具操作正常生效（放置/移除元素、预算增减）
   - 天数与事件日志同步推进，无跳变或回退
   - 反馈面板每轮正常更新

#### 场景三：自动推进中手动点击推进

1. 启动自动推进
2. 在自动推进间隔内快速点击「推进半日」按钮 2-3 次
3. **验证点**：
   - 不会出现重复推进或跳天（每 tick 仅增加 1）
   - 天数、事件日志、反馈面板三者保持同步
   - 自动推进定时器不受影响，继续正常运行

#### 场景四：自动推进中快速切换暂停/启动

1. 打开页面，确认自动推进处于关闭状态
2. 快速连续点击「自动推进」按钮 5-6 次（开-关-开-关...）
3. **验证点**：
   - 最终按钮状态与实际定时器状态一致（按钮文字、aria-pressed、视觉样式三者同步）
   - 不会出现「按钮显示暂停但实际不推进」或「按钮显示自动推进但实际在推进」的不一致
   - 多次快速切换后，天数不会异常跳增

#### 场景五：挑战完成时自动停止

1. 启动任意挑战（如「稳定守护者」）
2. 启动自动推进，等待挑战成功或失败
3. **验证点**：
   - 挑战成功/失败提示出现时，自动推进立即停止
   - 按钮文字恢复为「自动推进」，视觉样式恢复默认
   - 天数不再继续增加

#### 场景六：加载沙盘方案

1. 启动自动推进
2. 点击「沙盘方案」打开面板，确认自动推进已自动暂停
3. 选择任意已保存方案点击「加载」
4. **验证点**：
   - 关闭模态框后，自动推进保持暂停状态
   - 按钮显示「自动推进」，无脉冲动画
   - 加载的方案数据（天数、事件、稳定度等）完整显示，无旧定时器推进造成的错乱

#### 场景七：页面刷新/关闭清理

1. 启动自动推进
2. 刷新浏览器页面（F5 或 Cmd+R）
3. **验证点**：
   - 页面重新加载后，自动推进处于关闭状态
   - 按钮显示「自动推进」，背景为默认浅色
   - 无残留定时器继续运行

#### 无障碍验证

1. 使用浏览器开发者工具检查「自动推进」按钮
2. 启动推进，验证：
   - `aria-pressed="true"`
   - `aria-label="暂停自动推进"`
3. 停止推进，验证：
   - `aria-pressed="false"`
   - `aria-label="启动自动推进"`

### 数据存储说明

- 方案数据保存在浏览器的 `localStorage` 中，键名为 `tidepool_sandbox_scenarios`
- 清除浏览器数据或更换浏览器会导致方案丢失
- 方案按保存时间倒序排列，最新保存的排在最前
- 每个方案卡片显示：方案名称、保存时间、天数、稳定度、各物种数量概览

## Canvas 点击坐标验证说明

### 问题背景

在移动端或浏览器缩放后点击 Canvas 格子时，可能出现点击位置与实际选中格子不匹配的问题，尤其是画布被 CSS 拉伸后更明显。

### 修复原理

1. **坐标换算修正**：使用 `canvas.width / rect.width` 计算缩放比例，将 CSS 像素正确转换为 Canvas 实际像素坐标
2. **宽高比锁定**：CSS 中使用 `aspect-ratio: 920 / 620` 确保画布始终保持正确比例，避免变形
3. **响应式适配**：`max-width` 和 `max-height` 限制最大尺寸，`width: 100%` 确保小屏幕自适应

### 手动验证步骤

#### 桌面端验证

1. 打开 `index.html`，确保浏览器窗口宽度大于 980px（两栏布局）
2. 选择「岩缝」工具，点击画布左上角的第一个沙格子（第1行第2列，坐标约 x=90, y=80）
3. 验证：岩缝应准确出现在点击的格子上，不会偏移到相邻格子
4. 选择「清理」工具，点击刚才放置岩缝的格子
5. 验证：岩缝被正确清除，不会误清除其他格子

#### 浏览器缩放验证

1. 使用浏览器缩放功能（Ctrl+/- 或 Cmd+/-），分别测试 75%、100%、125%、150% 缩放比例
2. 在每个缩放比例下，点击画布四角的格子和中心格子
3. 验证：点击位置与选中格子始终一一对应，无偏移现象

#### 移动端/窄屏验证

1. 缩小浏览器窗口宽度至 640px 以下（移动端布局）
2. 选择「海藻」工具，依次点击第 1 行、第 5 行、第 9 行的中间格子
3. 验证：海藻准确放置在点击的格子中，不会出现横向或纵向偏移
4. 尝试点击画布边缘的格子（最左列、最右列、最上行、最下行）
5. 验证：边缘格子也能被正确选中，不会触发相邻格子

#### 窗口大小变化验证

1. 缓慢拖动浏览器窗口边缘，改变窗口宽度
2. 在拖动过程中停止，点击画布上的任意格子
3. 验证：窗口大小变化后，点击位置仍然准确对应正确的格子

### 验证通过标准

- 所有缩放比例下，点击位置与选中格子完全一致
- 桌面端和移动端布局下均无偏移
- 四角和边缘格子点击准确
- 快速点击和慢速点击均能正确响应
